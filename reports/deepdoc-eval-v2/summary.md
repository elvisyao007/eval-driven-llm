実施日: 2026-06-13  
対象: v1と同じ5件の日本語政府公開PDF（3ドキュメント使用: 02/03/04）  
環境: CPU only（embedding: ruri-v3-310m on CPU）, Python 3.12  
目的: v1の2つの限界を同時に収紧 — ① BM25専用 → 2×2比較追加, ② oracle 100% golden set → oracle <100% golden set v2

## 1. Golden Set v2: 難易度・Oracle Ceiling

### 1.1 設計方針（v1との比較）

| 項目 | v1 | v2 |
|------|----|----|
| 問数 | 20問 | 32問 |
| 難易度設計 | medium中心（直接的なキーワードマッチ） | medium 8問 + hard 20問 + oracle_fail 4問 |
| Oracle ceiling | 100%（20/20） | **87.5%（28/32）** |
| 検索器 | BM25のみ | BM25 + dense（ruri-v3）の2×2 |

### 1.2 Oracle Ceiling 検証（Sanity Gate）

```
Pipeline A oracle（plain全文検索）: 28/32 = 87.5%
Pipeline B oracle（DeepDoc全文検索）: 28/32 = 87.5%
Sanity gate: PASS（≥60%条件充足）
```

**Oracle-failing 4問**（正解キーワードがいずれのコーパスにも存在しない）:

| 問 | 質問 | 正解（計算値） | 不在理由 |
|----|------|----------------|----------|
| Q07 | 社会保障関係費+国債費の合計は？ | 664,957億円 | 和 = 2セルの算術；文書に記載なし |
| Q08 | 税収+その他収入の合計は？ | 868,925億円 | 和 = 2セルの算術；文書に記載なし |
| Q18 | 40代世帯主と40歳未満の消費支出差は？ | 51,192円 | 差 = 2行の算術；文書に記載なし |
| Q26 | 特別会計歳出-一般会計歳出は？ | 314.3兆円 | 差 = 2セルの算術；文書に記載なし |

**結論**: Oracle ceiling 87.5% は v1 の 100% より有意に低く、「正解キーワードを
持つ問い」だけでなく「文脈理解が必要な問い」を含む、より現実的なテストセット
であることが確認された。

## 3. 主要発見

### 3.1 DeepDoc の純価値: BM25 vs Dense の比較

**予想に反して、DeepDoc の優位性は Dense でより大きい（+25%）**。

| 検索器 | DeepDoc デルタ | 解釈 |
|--------|---------------|------|
| BM25 | +12.5% | v1(+15%)と同等。より難しい設問での再現性を確認 |
| Dense | +25.0% | **BM25より大きいデルタ。予想外の逆転** |

**なぜ Dense で差が拡大するのか**:

- Plain text（Pipeline A）は2,934チャンクを生成。多くは表から切り出された数字の断片
  （例: "3 01,646 272,214 312,758..."）。この断片は BM25 では数値マッチに使えるが、
  dense embedding では文脈なしの数字列として低品質なベクトルを生成する。
- DeepDoc（Pipeline B）は630チャンクを生成（4.7倍少ない）。構造化された
  テキストブロックと表テキストは、列ヘッダや文脈を保ちセマンティックに豊かな
  埋め込みを生成する。
- **結論**: Dense embedding は chunk 品質に敏感。DeepDoc のレイアウト理解は
  BM25 よりも Dense 検索で恩恵が大きい。

### 3.2 Plain × Dense の落とし込みポイント

A×dense（40.6%）がA×BM25（56.2%）より16ポイント低い理由:

| 問 | 失敗パターン |
|----|------------|
| Q01, Q03 | 数値は1文字の断片に存在するが、文脈なしの断片は semantic に類似せず |
| Q05, Q19 | 表中の具体的数値は dense では行コンテキストがないと見つけられない |
| Q28, Q32 | 財務諸表の深部の数値は Plain の断片化で埋め込みが弱い |

### 3.3 両検索器とも難しい問

以下の問は4組合せすべてで失敗（除 oracle-fail）:

- **Q09, Q12, Q17, Q20**: 月別消費支出の特定値・年齢別消費の深い表セル
  - Q09（2月消費272,214）: 表テキストに存在するが、表0の月別行から取り出しにくい
  - Q12（70代消費249,177）: 両パイプラインとも年齢別表セルを top-5 に入れられない
  - Q17（60代消費306,476）: 同上
  - Q20（65歳以上261,235）: 同上

これらは「1行の表セル内の具体的数値」で、テーブル全体をひとつのチャンクとして扱う
現行設計では BM25/dense 問わず取りこぼしやすい。セル単位チャンク化が改善策候補。

## 4. v1 との比較（収紧の確認）

| 評価 | 検索器 | Pipeline A | Pipeline B | Delta |
|------|--------|-----------|-----------|-------|
| **v1** | BM25 | 75%（20問,oracle100%） | 90%（20問,oracle100%） | +15% |
| **v2** | BM25 | 56.2%（32問,oracle87.5%） | 68.8% | +12.5% |
| **v2** | Dense | 40.6% | 65.6% | +25.0% |

**v1→v2 での変化の解釈**:
- BM25 デルタが +15% → +12.5% に縮小: より難しい問セットで DeepDoc の優位性は
  概ね維持されている。oracle 100% の v1 に比べ v2 は harder、数値は下がるが傾向は一貫。
- Dense の +25% は新発見。v1 では測定されていなかった。

## 5. 誠実さの限界表明

- 3ドキュメント・32問（うち oracle-fail 4問除外すると実質28問）は **初期信号**。定説として扱わない。
- 同一の chunk_size=300/step=150 設計を BM25・Dense 両方で使用。Dense に最適な
  chunk 設計を試みていない（通常は小さい chunk が Dense に有利）。
- ruri-v3-310m は日本語特化だが、本評価は chunk の質差による影響と embedder
  の能力を分離していない。
- GPU 未使用（CPU embedding; 本番環境では GPU で 10-30× 高速化予定）。
- **v2の結論はv1を収紧するもの、覆すものではない**:
  - v1: BM25 で DeepDoc +15% (oracle=100%の易しいセット)
  - v2: BM25 で+12.5%, Dense で+25% (oracle=87.5%のより難しいセット)
  - 両評価とも「埋め込みフォント PDF でのレイアウト解析は検索精度を向上させる」
    という方向性は一致している。
- **密度追加の新知見**: Dense でのデルタがより大きい点は追加調査に値するが、
  chunk 設計・embedding モデル選択・評価問数の三重の交絡因子がある。

## 6. Embedding モデル選択の記録

**使用モデル**: `cl-nagoya/ruri-v3-310m`  
**選択理由**:
1. 本プロジェクトの主要パイプライン（DECISIONS.md ADR-0007）で既採用の標準モデル
2. 日本語文章理解に特化（名古屋大学/NEDO 開発）、日本語 IR ベンチマークで高性能
3. すでに `/mnt/cache/hf` にキャッシュ済み（追加ダウンロード不要）
4. sentence-transformers インターフェース経由で `.venv` から利用可能

bge-m3（Ollama 経由）は多言語対応だが、ruri-v3 は日本語専用設計で
本評価の日本語文書に対して適切な選択。Dense 検索の基準として本プロジェクトの
既定 embedder を使うことで、v2 の dense 結果は本番パイプラインとの
直接比較が可能になる。
