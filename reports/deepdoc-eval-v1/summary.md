# DeepDoc Eval v1 — 結果サマリー

実施日: 2026-06-12  
対象: 5件の日本語政府公開PDF (計119ページ)  
環境: CPU only, Python 3.12, DeepDoc PdfParser API  
誠実さの注記: 5ドキュメント・20問は初期信号であり、定説ではない。
生産決定には追加サンプルと多様なドキュメント種別での再現確認を要する。

---

## 1. 解析品質

### 1.1 OCR 系統的誤認識

**重要発見: 誤認識はテキスト抽出経路によって二分される。**

| サンプル | フォント種別 | テキスト抽出経路 | 令和参照数 | 令→今/命 誤り | 誤り率 |
|---------|------------|---------------|----------|------------|--------|
| 01_nta_kakutei (確定申告書) | 複合フォーム/OCRフォールバック | OCR (pdfplumber失敗→OCR) | 2 | 2 | **100%** |
| 02_mof_budget (予算概要) | 埋め込みフォント | pdfplumber ネイティブ | 10 | 0 | 0% |
| 03_stat_kakei (家計調査) | 埋め込みフォント | pdfplumber ネイティブ | 2 | 0 | 0% |
| 04_mof_fiscal (財政資料) | 埋め込みフォント | pdfplumber ネイティブ | 62 | 0 | 0% |
| 05_archives_scan (公文書館) | 混在(活字) | pdfplumber ネイティブ | 0 | 0 | N/A |

**解釈**:
- 令→今/命 の誤認識は **OCR フォールバック経路でのみ発生**する。
  埋め込みテキストを持つ近代的な PDF (02-04) では pdfplumber のネイティブ
  テキスト抽出が使われ、年号はゼロエラーで取得される。
- 複雑なフォームフォントを持つ確定申告書 (01) は全ページが OCR フォールバック
  となり、年号参照 2/2 が誤認識 (100%)。
- **リスクが高いのは「フォーム型」や「スキャン」PDF**。行政 DX で多用される
  e-Tax 様式がこれに該当する。埋め込みフォントの多くの帳票 PDF は安全。

### 1.2 TSR (表構造認識) 精度

対象: sample 01 (確定申告書、複雑格子フォーム)  
方法: 20セルの人工グランドトゥルース vs DeepDoc 抽出テキスト  

| 指標 | 値 |
|------|---|
| グランドトゥルース合計 | 20セル |
| 完全一致 | 6/20 = **30%** |
| 一致+既知誤字許容 | 7/20 = **35%** |
| 未検出 | 13/20 |

**解釈**:
- 税務申告書フォームでは 30% と低精度。主因は「埋め込みフォントが garbled
  化 → OCR 頼り → 認識精度が落ちる」という連鎖。
- テーブルから抽出できたセルは「個人番号」「生年月日」「氏名」「職業」
  「公的年金等」「扶養控除」など、比較的単純なラベル。
- 複合構造 (結合セル、小区画の連続) に弱い。

### 1.3 解析速度 (CPU, モデルキャッシュ済み)

| サンプル | ページ数 | DeepDoc 秒 | 秒/ページ | Plain 秒 |
|---------|---------|------------|---------|---------|
| 01_nta (確定申告書) | 4p | 138s | 34.5s | 0.2s |
| 02_mof_budget (予算) | 14p | 25s | 1.8s | 0.2s |
| 03_stat_kakei (家計調査) | 24p | 221s | 9.2s | 0.6s |
| 04_mof_fiscal (財政) | 68p | 474s | 7.0s | 2.6s |
| 05_archives_scan | 9p | 37s | 4.1s | 0.1s |

**注**: ページ毎処理時間の幅が大きい。表が多いページ(家計調査・財政資料)は
OCR + TSR の両方が走るため高コスト。スキャン文書でない限り、
CPU でのバッチ処理は 5-10s/page 程度を見込む必要がある。
**GPU 化 (RTX 5090) で 10-30× の高速化が見込まれ、Phase 2 推奨。**

## 2. 下流検索デルタ

### 2.1 評価設定

| 項目 | 値 |
|------|---|
| ゴールデンセット | 20問 (Q01-Q20) |
| 対象ドキュメント | sample 02, 03, 04 (埋め込みフォント PDF) |
| Pipeline A | pdfplumber plain text extraction → BM25 |
| Pipeline B | DeepDoc PdfParser (layout + TSR) → BM25 |
| 指標 | hit@5 (answer_keywords が top-5 チャンクに含まれるか) |
| サニティゲート | Oracle 全文検索でアンサーが corpus 内にあることを確認 |

### 2.2 サニティゲート

- Pipeline A oracle (全文): 20/20 = 100% ✅  
- Pipeline B oracle (全文): 20/20 = 100% ✅  
- **PASS** — 両コーパスとも回答が存在する。指標は信頼できる。

### 2.3 hit@5 結果

| Pipeline | hit@5 | 問数 |
|---------|-------|-----|
| **A: plain pdfplumber** | **75%** | 15/20 |
| **B: DeepDoc PdfParser** | **90%** | 18/20 |
| **Delta (B - A)** | **+15%** | +3問 |

DeepDoc が A に勝った設問: Q04, Q09, Q13  
どちらも外した設問: Q05, Q11

**DeepDoc が取りこぼした理由の分析:**
- Q05 (公債金 286,490億円): 数値が表内に埋まっており、チャンク境界で分断。
  A も外す → 表構造問題。
- Q11 (目次の章数): 数え上げが必要な推論問題。単純 BM25 では解けない。

**DeepDoc が A を上回った理由:**
- Q04 (税収 784,400億円): plain が 639 チャンクに冗長な行分割で埋もれたが、
  DeepDoc は 63 チャンクに凝縮したため正しい数値行が top-5 内に入った。
- Q09 (二人以上の世帯): plain のランダム行分割でキーワードが分断。
  DeepDoc は段落単位のチャンクで文脈を保持。
- Q13 (令和7年10月): DeepDoc がヘッダー/タイトルを独立ブロックとして抽出。
  plain では目次の長文行に埋もれた。

## 3. 判断: DeepDoc の日文における純価値

### 利用すべきシナリオ

| シナリオ | DeepDoc の価値 | 備考 |
|---------|-------------|------|
| 埋め込みフォント PDF (政府資料、ビジネス文書) | **高い** | +15% hit@5。レイアウト分離が検索精度を上げる |
| 表重視の問い (予算表、統計表) | **中〜高い** | TSR で表をチャンクに取り込む (plain は行列が混線) |
| スキャン PDF / 古文書 | **要注意** | OCR 経路で年号・特定漢字の誤認識 ~100%。後処理辞書が必要 |
| 複雑なフォーム型 PDF (e-Tax 申告書など) | **低い** | 1チャンクしか抽出できない事例あり。専用パーサを検討 |

### OCR 誤認識が dealbreaker になる条件

1. **検索キーが年号**: 「令和7年度」を検索するのに「今和7年」になっていると完全ミスになる。
   → 対策: 後処理辞書 (令和年号パターンの正規化) を挟む。
2. **スキャン PDF がコアドキュメント**: Phase 1 で発見した OCR 誤認識は
   スキャン/OCRフォールバック限定。企業文書の多くは埋め込みフォントのため対象外。
3. **完全な表セル内容に依存する問い**: TSR 精度が 30-35% (フォーム型) では
   表セルの正確な値を検索で引きたい場合に不十分。

### 現段階での推奨

- **埋め込みフォント PDF のコーポラス**: DeepDoc PdfParser を採用。
  hit@5 で plain + 15% はレイアウト理解の有効性を示す初期信号。
- **スキャン/フォーム PDF**: Phase 2 でさらに検証。現状は年号正規化後処理を条件に採用検討可。
- **GPU 化**: CPU 7-34s/page は本番バッチには遅い。Phase 2 で GPU ベンチ必須。
- **全量 CER (文字誤り率) 計測**: 本 eval は抽象的なキーワードマッチ。
  法律用語・固有名詞の正確な CER 計測は v2 スコープ。

## 4. 測定限界の表明

- ゴールデンセット 20問・3ドキュメントは **初期信号**。定説として扱わない。
- BM25 + bigram tokenizer は日本語 IR の baseline。dense retrieval との
  比較は Phase 2。
- TSR 精度の ground truth は 20セルのみで統計的に粗い。
- 全量 CER 計測は行っていない (範囲外)。
- GPU 速度は未測定 (Phase 2 推奨)。
